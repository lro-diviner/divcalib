116 C From Leonard J. Moss of SLAC:
117 
118 C Here's a hybrid QuickSort I wrote a number of years ago. It's
119 C based on suggestions in Knuth, Volume 3, and performs much better
120 C than a pure QuickSort on short or partially ordered input arrays.
121 
122  SUBROUTINE sortrx(N,DATA,MYINDEX)
123 C===================================================================
124 C
125 C SORTRX -- SORT, Real input, myindex output
126 C
127 C
128 C Input: N INTEGER
129 C DATA REAL
130 C
131 C Output: MYINDEX INTEGER (DIMENSION N)
132 C
133 C This routine performs an in-memory sort of the first N elements of
134 C array DATA, returning into array MYINDEX the indices of elements of
135 C DATA arranged in ascending order. Thus,
136 C
137 C DATA(MYINDEX(1)) will be the smallest number in array DATA;
138 C DATA(MYINDEX(N)) will be the largest number in DATA.
139 C
140 C The original data is not physically rearranged. The original order
141 C of equal input values is not necessarily preserved.
142 C
143 C===================================================================
144 C
145 C SORTRX uses a hybrid QuickSort algorithm, based on several
146 C suggestions in Knuth, Volume 3, Section 5.2.2. In particular, the
147 C "pivot key" [my term] for dividing each subsequence is chosen to be
148 C the median of the first, last, and middle values of the subsequence;
149 C and the QuickSort is cut off when a subsequence has 9 or fewer
150 C elements, and a straight insertion sort of the entire array is done
151 C at the end. The result is comparable to a pure insertion sort for
152 C very short arrays, and very fast for very large arrays (of order 12
153 C micro-sec/element on the 3081K for arrays of 10K elements). It is
154 C also not subject to the poor performance of the pure QuickSort on
155 C partially ordered data.
156 C
157 C Created: 15 Jul 1986 Len Moss
158 C
159 C===================================================================
160 
161  implicit real*8 (a-h,o-z)
162  INTEGER n,myindex(n)
163  REAL*8 data(n)
164 
165  INTEGER lstk(31),rstk(31),istk
166  INTEGER l,r,i,j,p,myindexp,myindext
167  REAL*8 datap
168 
169 C QuickSort Cutoff
170 C
171 C Quit QuickSort-ing when a subsequence contains M or fewer
172 C elements and finish off at end with straight insertion sort.
173 C According to Knuth, V.3, the optimum value of M is around 9.
174 
175  INTEGER m
176  parameter(m=9)
177 
178 C===================================================================
179 C
180 C Make initial guess for MYINDEX
181 
182  DO 50 i=1,n
183  myindex(i)=i
184  50 continue
185 
186 C If array is short, skip QuickSort and go directly to
187 C the straight insertion sort.
188 
189  IF (n.LE.m) goto 900

191 C===================================================================
192 C
193 C QuickSort
194 C
195 C The "Qn:"s correspond roughly to steps in Algorithm Q,
196 C Knuth, V.3, PP.116-117, modified to select the median
197 C of the first, last, and middle elements as the "pivot
198 C key" (in Knuth's notation, "K"). Also modified to leave
199 C data in place and produce an MYINDEX array. To simplify
200 C comments, let DATA[I]=DATA(MYINDEX(I)).
201 
202 C Q1: Initialize
203  istk=0
204  l=1
205  r=n
206 
207  200 continue
208 
209 C Q2: Sort the subsequence DATA[L]..DATA[R].
210 C
211 C At this point, DATA[l] <= DATA[m] <= DATA[r] for all l < L,
212 C r > R, and L <= m <= R. (First time through, there is no
213 C DATA for l < L or r > R.)
214 
215  i=l
216  j=r
217 
218 C Q2.5: Select pivot key
219 C
220 C Let the pivot, P, be the midpoint of this subsequence,
221 C P=(L+R)/2; then rearrange MYINDEX(L), MYINDEX(P), and MYINDEX(R)
222 C so the corresponding DATA values are in increasing order.
223 C The pivot key, DATAP, is then DATA[P].
224 
225  p=(l+r)/2
226  myindexp=myindex(p)
227  datap=DATA(myindexp)
228 
229  IF (DATA(myindex(l)) .GT. datap) THEN
230  myindex(p)=myindex(l)
231  myindex(l)=myindexp
232  myindexp=myindex(p)
233  datap=DATA(myindexp)
234  ENDIF
235 
236  IF (datap .GT. DATA(myindex(r))) THEN
237  IF (DATA(myindex(l)) .GT. DATA(myindex(r))) THEN
238  myindex(p)=myindex(l)
239  myindex(l)=myindex(r)
240  ELSE
241  myindex(p)=myindex(r)
242  ENDIF
243  myindex(r)=myindexp
244  myindexp=myindex(p)
245  datap=DATA(myindexp)
246  ENDIF
247 
248 C Now we swap values between the right and left sides and/or
249 C move DATAP until all smaller values are on the left and all
250 C larger values are on the right. Neither the left or right
251 C side will be internally ordered yet; however, DATAP will be
252 C in its final position.
253 
254  300 continue
255 
256 C Q3: Search for datum on left >= DATAP
257 C
258 C At this point, DATA[L] <= DATAP. We can therefore start scanning
259 C up from L, looking for a value >= DATAP (this scan is guaranteed
260 C to terminate since we initially placed DATAP near the middle of
261 C the subsequence).
262 
263  i=i+1
264  IF (DATA(myindex(i)).LT.datap) goto 300
265

268 C Q4: Search for datum on right <= DATAP
269 C
270 C At this point, DATA[R] >= DATAP. We can therefore start scanning
271 C down from R, looking for a value <= DATAP (this scan is guaranteed
272 C to terminate since we initially placed DATAP near the middle of
273 C the subsequence).
274 
275  j=j-1
276  IF (DATA(myindex(j)).GT.datap) goto 400
277 
278 C Q5: Have the two scans collided?
279 
280  IF (i.LT.j) THEN
281 
282 C Q6: No, interchange DATA[I] <--> DATA[J] and continue
283 
284  myindext=myindex(i)
285  myindex(i)=myindex(j)
286  myindex(j)=myindext

288  ELSE
289 
290 C Q7: Yes, select next subsequence to sort
291 C
292 C At this point, I >= J and DATA[l] <= DATA[I] == DATAP <= DATA[r],
293 C for all L <= l < I and J < r <= R. If both subsequences are
294 C more than M elements long, push the longer one on the stack and
295 C go back to QuickSort the shorter; if only one is more than M
296 C elements long, go back and QuickSort it; otherwise, pop a
297 C subsequence off the stack and QuickSort it.
298 
299  IF (r-j .GE. i-l .AND. i-l .GT. m) THEN
300  istk=istk+1
301  lstk(istk)=j+1
302  rstk(istk)=r
303  r=i-1
304  ELSE IF (i-l .GT. r-j .AND. r-j .GT. m) THEN
305  istk=istk+1
306  lstk(istk)=l
307  rstk(istk)=i-1
308  l=j+1
309  ELSE IF (r-j .GT. m) THEN
310  l=j+1
311  ELSE IF (i-l .GT. m) THEN
312  r=i-1
313  ELSE
314 C Q8: Pop the stack, or terminate QuickSort if empty
315  IF (istk.LT.1) goto 900
316  l=lstk(istk)
317  r=rstk(istk)
318  istk=istk-1
319  ENDIF
320  goto 200
321  ENDIF
322 
323  900 continue
